# 桌面应用重启式打开设计

## 概述

当前“应用管理”中的“打开”操作会直接启动目标应用。对于桌面应用，这会导致已经运行的实例继续保留旧状态，而新写入的代理配置和模型配置不一定立刻生效。

本次改动将桌面应用的“打开”语义调整为“重启式打开”：

1. 仅针对 `Codex Desktop` 和 `Claude Desktop`
2. 启动前先检测应用是否已运行
3. 若已运行，则先请求退出并等待进程结束
4. 确认退出后，再重新启动应用

CLI 应用不受影响，仍维持现有启动方式。

## 目标

- 确保桌面应用每次通过 AI Proxy 打开时，都会读取最新配置
- 保持前端交互不变，用户仍然只点击一次“打开”
- 将行为收敛在后端启动链路中，避免前端增加额外状态编排

## 非目标

- 不为 CLI 应用增加“已运行则先退出再启动”的行为
- 不引入新的前端 API 端点
- 不跟踪桌面应用的长期运行状态，只在启动瞬间做一次检查和处理

## 方案对比

### 方案 A：后端内聚处理（采用）

在现有 `POST /api/apps/launch` 流程内部，对桌面应用增加“检查运行状态 -> 优雅退出 -> 等待退出 -> 启动”的逻辑。

优点：

- 前端无须改协议
- 行为集中，避免重复逻辑
- 更容易保证写配置与重启流程的一致性

缺点：

- 需要在后端实现少量平台相关逻辑

### 方案 B：前端编排多个接口

前端先查询运行状态，必要时调用退出接口，最后再调启动接口。

不采用原因：

- 需要新增接口
- 增加前后端耦合和竞态风险
- 对当前场景属于过度设计

### 方案 C：不检测运行状态，直接使用系统“重新打开”命令

不采用原因：

- 各平台支持和行为不一致
- 难以明确区分“已退出”与“仍在运行”
- 错误反馈不稳定

## 详细设计

### 调整范围

后端文件：

- `src-tauri/src/apps/launcher.rs`

必要时补充测试：

- `src-tauri/src/apps/launcher.rs` 内部单测

前端文件无协议变化，`src/views/Apps.vue` 保持现有调用方式。

### 启动流程

`launch()` 的总体分支保持不变：

- CLI 应用继续走 `launch_cli()`
- 桌面应用继续走 `launch_desktop()`，但其内部改为“重启式打开”

新的桌面启动流程如下：

1. 根据 `install_path` 和 `AppType` 解析出应用标识
2. 检查该桌面应用是否正在运行
3. 若未运行，直接执行现有启动逻辑
4. 若正在运行，先请求优雅退出
5. 在限定时间内轮询确认进程已结束
6. 退出完成后，再执行启动命令

### 平台策略

#### macOS

- 运行检测：使用 `pgrep` 或等价方式按应用名检测
- 退出方式：优先使用 `osascript -e 'tell application "AppName" to quit'`
- 启动方式：继续使用 `open <install_path>`

之所以优先使用 AppleScript，是因为 `.app` 应用通常支持正常退出流程，能尽量避免数据损坏或状态异常。

#### Windows / Linux

- 当前仓库已包含跨平台桌面启动分支，但本次需求的明确场景来自 desktop 应用卡片行为
- 第一阶段只保证 macOS 的“先退出再打开”逻辑完整生效
- 对于其他平台，保持原有直接启动逻辑，不额外引入高风险进程控制

这样可以在当前用户环境下先解决实际问题，同时避免在缺少验证条件时引入脆弱实现。

## 错误处理

- 运行状态检测失败：记录日志，并按“未运行”处理，直接尝试启动
- 退出命令执行失败：返回启动失败，提示应用退出失败
- 等待退出超时：返回启动失败，提示应用仍在运行，避免重复拉起实例
- 启动命令失败：保持现有错误返回逻辑

选择“检测失败继续启动”是为了尽量容忍偶发系统命令问题；选择“退出失败或超时即终止启动”是为了避免旧实例未退出时继续启动导致状态不一致。

## 接口影响

不新增 API，不修改前端请求体，不修改响应结构。

`POST /api/apps/launch` 的业务语义变为：

- CLI：启动
- Desktop：必要时先退出旧实例，再启动

## 测试策略

优先补充决策级单测，而非依赖真实系统环境的集成测试：

- CLI 应用仍走原有分支
- Desktop 应用进入重启式分支
- 应用名称解析逻辑正确
- 超时常量和轮询逻辑具备可测试的辅助函数边界

不在单测中直接依赖真实 `osascript`、`pgrep` 或真实桌面应用实例。

## 风险与约束

- 桌面应用名称与真实进程名可能存在平台差异，因此应用名解析需要保持显式映射
- 某些应用可能在收到退出命令后短时间内仍保留后台进程，因此必须等待进程真正结束
- 若应用存在未保存内容，优雅退出可能触发应用自身的保存流程或阻塞，这属于桌面应用自身行为

## 成功标准

- 在 macOS 上点击 `Codex Desktop` 或 `Claude Desktop` 的“打开”按钮时：
  - 若应用未运行，正常启动
  - 若应用已运行，先退出旧实例，再重新启动
- CLI 应用行为不变
- 前端无额外按钮、无额外确认步骤
